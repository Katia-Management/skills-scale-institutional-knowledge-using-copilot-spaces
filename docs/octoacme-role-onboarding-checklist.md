# OctoAcme Role Onboarding Checklist

This checklist supports teams in successfully onboarding new team members to their roles at OctoAcme. Use this template to ensure new hires have the context, access, and relationships they need to be effective.

---

## General Onboarding (All Roles)

### Week 1: Orientation and Access
- [ ] Complete HR and administrative onboarding
- [ ] Set up development environment and tools
- [ ] Access granted to required systems (GitHub, project boards, communication tools)
- [ ] Review [Project Management Overview](octoacme-project-management-overview.md)
- [ ] Read [Roles and Personas](octoacme-roles-and-personas.md) - understand your role and how you interact with others
- [ ] Introduction meeting with direct manager
- [ ] Meet the team in daily standup or team meeting

### Week 2: Context Building
- [ ] Review current project documentation in the docs/ folder
- [ ] Shadow team members in your role (if available)
- [ ] Attend relevant recurring meetings (standups, planning, reviews)
- [ ] Identify a mentor or buddy for questions
- [ ] Review recent retrospectives to understand team learnings

### Week 3-4: Active Contribution
- [ ] Take on first small task or assignment
- [ ] Participate actively in team ceremonies
- [ ] Begin building relationships with cross-functional partners
- [ ] Provide onboarding feedback to improve the process

## Role-Specific Onboarding

### For Developers
- [ ] Set up local development environment and run the application
- [ ] Review coding standards and style guides
- [ ] Understand CI/CD pipeline and deployment process
- [ ] Review [Execution and Tracking](octoacme-execution-and-tracking.md) for workflow practices
- [ ] Complete first pull request with code review
- [ ] Understand Definition of Done and quality gates
- [ ] Meet with Engineering Lead for technical architecture overview

### For Product Managers
- [ ] Review product vision, roadmap, and current priorities
- [ ] Understand customer personas and user research insights
- [ ] Review success metrics and dashboards
- [ ] Meet with Data Analyst to understand available metrics
- [ ] Shadow customer research sessions or review recordings
- [ ] Review [Project Initiation](octoacme-project-initiation.md) for one-pager format
- [ ] Meet with key stakeholders to understand their needs

### For Project Managers
- [ ] Review active projects, timelines, and dependencies
- [ ] Understand risk register format and escalation paths
- [ ] Review [Risks and Communication](octoacme-risks-and-communication.md)
- [ ] Meet with stakeholders to understand communication preferences
- [ ] Shadow existing PM in facilitation of ceremonies
- [ ] Review project board workflows and status reporting formats
- [ ] Access to project tracking and reporting tools

### For QA Engineers / Testers
- [ ] Understand test strategy and automation frameworks
- [ ] Review Definition of Done and acceptance criteria standards
- [ ] Set up test environment and tools
- [ ] Review [Execution and Tracking](octoacme-execution-and-tracking.md) for quality gates
- [ ] Shadow testing workflow from backlog to QA sign-off
- [ ] Understand bug reporting and tracking processes
- [ ] Meet with developers to discuss testability practices

### For Engineering Lead / Technical Lead
- [ ] Review architecture documentation and technical decisions
- [ ] Understand current technical debt and priorities
- [ ] Meet with developers individually for context and needs
- [ ] Review code review standards and practices
- [ ] Understand technology stack, tooling, and patterns
- [ ] Identify technical risks and improvement opportunities
- [ ] Meet with Product Manager for roadmap and feature planning context

### For Release Manager
- [ ] Review [Release and Deployment](octoacme-release-and-deployment.md)
- [ ] Understand release calendar and deployment windows
- [ ] Access to production systems and deployment tools
- [ ] Review rollback procedures and incident playbooks
- [ ] Meet with DevOps Engineer for infrastructure context
- [ ] Shadow a release deployment end-to-end
- [ ] Review release notes format and communication templates

### For Security Engineer
- [ ] Review security standards, policies, and compliance requirements
- [ ] Understand security scanning tools and CI integration
- [ ] Access to vulnerability tracking and reporting systems
- [ ] Review recent security findings and remediation status
- [ ] Meet with Engineering Lead for architecture security review
- [ ] Understand incident response procedures for security issues
- [ ] Review secure coding guidelines and training materials

### For Data Analyst / Metrics Lead
- [ ] Access to analytics platforms and data sources
- [ ] Review existing dashboards and KPIs
- [ ] Understand data collection and instrumentation approach
- [ ] Meet with Product Manager to understand key questions and metrics needs
- [ ] Review data governance and privacy requirements
- [ ] Understand experiment and A/B testing framework (if applicable)
- [ ] Document current data gaps and improvement opportunities

### For UX Researcher / Customer Insights Specialist
- [ ] Review existing user research and customer feedback
- [ ] Understand customer personas and journey maps
- [ ] Access to user research tools and repositories
- [ ] Meet with Product Manager for research priorities
- [ ] Review customer feedback channels (support, surveys, etc.)
- [ ] Understand research ethics and privacy requirements
- [ ] Plan first research initiative or usability study

### For DevOps Engineer
- [ ] Access to cloud infrastructure and CI/CD systems
- [ ] Review infrastructure-as-code repositories
- [ ] Understand monitoring, logging, and alerting setup
- [ ] Review runbooks and incident response procedures
- [ ] Meet with developers to understand infrastructure needs
- [ ] Review security and compliance requirements for infrastructure
- [ ] Understand deployment procedures and release schedules
- [ ] Identify automation and infrastructure improvement opportunities

### For Business Analyst / Requirements Analyst
- [ ] Review requirements documentation format and templates
- [ ] Understand stakeholder landscape and key contacts
- [ ] Access to requirements management tools
- [ ] Review [Project Planning](octoacme-project-planning.md) for backlog creation
- [ ] Shadow requirements gathering sessions
- [ ] Understand dependency mapping and impact analysis practices
- [ ] Meet with Product Manager and Project Manager for context on priorities

### For Knowledge Manager / Documentation Lead
- [ ] Review all process documentation in docs/ folder
- [ ] Understand documentation standards and formats
- [ ] Access to knowledge repositories and wikis
- [ ] Review recent retrospective learnings and action items
- [ ] Identify documentation gaps and outdated content
- [ ] Understand Copilot Spaces integration for knowledge access
- [ ] Meet with all role representatives to understand documentation needs
- [ ] Create documentation improvement plan

### For Stakeholders
- [ ] Introduction to project purpose and expected outcomes
- [ ] Review project timeline and key milestones
- [ ] Understand communication cadence (updates, reviews, demos)
- [ ] Clarify decision-making authority and input opportunities
- [ ] Meet with Project Manager for status and escalation processes
- [ ] Review project artifacts (charter, roadmap, risk register)
- [ ] Understand feedback channels and how to raise concerns

## 30-60-90 Day Goals Template

Use this template to set clear expectations for the first 90 days:

### 30 Days
- [ ] Complete all onboarding activities
- [ ] Understand team processes and workflows
- [ ] Make first meaningful contributions
- [ ] Build initial relationships with key partners

### 60 Days
- [ ] Independently complete work in your role
- [ ] Identify and propose at least one improvement
- [ ] Actively participate in team ceremonies and decision-making
- [ ] Build deeper expertise in domain and tools

### 90 Days
- [ ] Full productivity in role responsibilities
- [ ] Mentor or support onboarding of newer team members
- [ ] Contribute to process improvements or retrospectives
- [ ] Demonstrate impact through measurable outcomes

## Onboarding Feedback

At the end of your onboarding period, please provide feedback to help us improve:

1. What aspects of onboarding were most valuable?
2. What was missing or could be improved?
3. How well did the role description match reality?
4. What would have helped you ramp up faster?
5. What documentation or resources were most/least useful?

Use the [Add Content to Project Management Process Docs](../.github/ISSUE_TEMPLATE/add-update-content-to-process-docs.yml) template to submit improvement suggestions, or provide feedback directly to your manager or the Knowledge Manager.
